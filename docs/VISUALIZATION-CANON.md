## Философия визуализации

Это **не визуализация данных** — это **визуализация состояния поля**.

### Что это НЕ

- ❌ Не граф с метриками (degree, centrality, PageRank)
- ❌ Не дашборд с числами
- ❌ Не рейтинг участников
- ❌ Не аналитика поведения
- ❌ Не социальная сеть

### Что это ЕСТЬ

- ✅ Отражение присутствия (кто здесь, как давно)
- ✅ Геометрия связей (форма, не количество)
- ✅ Ритм дыхания (эмиссия → передача → признание → сгорание)
- ✅ Поле смысла (где утолщается признание)
- ✅ Живое пространство (не застывший граф)

## Источники данных

### 1. acts_log → ro.DAG edges

**Источник истины:** `acts_log` (SSOT)

**Типы актов:**
- `EMISSION` — рождение импульса (6 У.Е.)
- `TRANSFER` — передача импульса (связь между О.К.)
- `RECOGNITION` — признание смысла (утолщение связи)
- `BURNED` — сгорание (освобождение)

**Рёбра графа:** `ro_dag_edges`
- `FLOW` — передача У.Е. (TRANSFER)
- `CAUSAL` — причинная связь (BURNED → EMISSION)
- `RECOGNITION` — признание (У.М. → акт)

**Метаданные рёбер:**
- `witness_scope` — глубина участия (personal|observation|indirect)
- `triad_link` — область ценности (T1-T5)
- `marker_type` — тип признания (value|gratitude|alignment|witness|support)

### 2. acts_annotations → У.М.

**Слой признания:** `acts_annotations` с типом `RECOGNITION`

**Payload:**
```json
{
  "marker_type": "value",
  "marker_text": "Вклад в общее дело",
  "witness_scope": "observation",
  "triad_link": "T1"
}
```

**Связь с графом:**
- Каждый У.М. создаёт ребро RECOGNITION в ro.DAG
- Утолщает связь между О.К., не добавляет число

### 3. directions → векторы

**Источник:** `/api/mirror` → `directions`

**Структура:**
```json
{
  "given": { "count": 8, "direction": "outward" },
  "received": { "count": 5, "direction": "inward" },
  "burned": { "count": 3, "direction": "release" }
}
```

**Важно:** Это **независимые векторы**, не скаляр.

### 4. field → насыщенность

**Источник:** `/api/mirror` → `field`

**Структура:**
```json
{
  "connections": "sparse|woven|dense",
  "activity": "quiet|active|resonant"
}
```

**Правила:**
- `connections` — качество связности (не количество связей)
- `activity` — качество активности (не частота действий)

## Правила трансформации (данные → форма)

### Узел = О.К.

**Визуальные свойства:**
- **Размер:** НЕ зависит от "важности" или "рейтинга"
- **Цвет:** Отражает `field.activity` (quiet → тусклый, resonant → яркий)
- **Пульсация:** Отражает `trace.silence_hours` (недавняя активность → быстрая пульсация)
- **Свечение:** Отражает `presence.ue_flow` (есть У.Е. → мягкое свечение)

**Запрещено:**
- Делать узлы больше/меньше на основе COUNT(*)
- Показывать "топ-10" или "самые активные"
- Сравнивать узлы между собой

### Импульс узла = последняя активность

**Источник:** `trace.last_act_at`, `trace.silence_hours`

**Визуализация:**
- **Time decay:** Чем дольше молчание, тем тусклее узел
- **Не исчезновение:** Узел не пропадает, а затухает
- **Порог:** После 7 дней молчания — минимальная яркость (но не 0)

**Формула (пример):**
```javascript
const opacity = Math.max(0.2, 1 - (silence_hours / (7 * 24)));
```

### Толщина связи = количество refs

**Источник:** `ro_dag_edges` → количество рёбер между двумя О.К.

**Визуализация:**
- **1 ребро:** Тонкая линия (базовая связь)
- **2-5 рёбер:** Средняя линия (повторяющаяся связь)
- **6+ рёбер:** Толстая линия (устойчивая связь)

**Запрещено:**
- Показывать число рёбер как текст
- Делать "самую толстую связь" выделенной
- Агрегировать в "силу связи"

### У.М. → локальное свечение

**Источник:** `recognition.recent_markers`, `recognition_field`

**Визуализация:**
- **Не бейдж:** У.М. не висит как иконка на узле
- **Локальное свечение:** Ребро или узел светится в месте признания
- **Цвет свечения:** Зависит от `marker_type`:
  - `value` → золотистый
  - `gratitude` → тёплый оранжевый
  - `alignment` → синий
  - `witness` → серебристый
  - `support` → зелёный

**Интенсивность свечения:**
- Зависит от `witness_scope`:
  - `personal` → яркое свечение
  - `observation` → среднее свечение
  - `indirect` → слабое свечение

**Запрещено:**
- Показывать COUNT(У.М.) как число
- Делать "рейтинг по признанию"
- Агрегировать У.М. в score

### Directions → геометрия движения

**Источник:** `directions`

**Визуализация:**

#### given (outward) — расширение наружу
- **Форма:** Узел излучает импульсы наружу (частицы, волны)
- **Направление:** От узла к другим узлам
- **Интенсивность:** Зависит от `directions.given.count` (но не показывается как число)

#### received (inward) — притяжение к центру
- **Форма:** Узел притягивает импульсы к себе (воронка, гравитация)
- **Направление:** От других узлов к узлу
- **Интенсивность:** Зависит от `directions.received.count`

#### burned (release) — рассеивание
- **Форма:** Узел рассеивает импульсы (дым, туман, растворение)
- **Направление:** От узла в пространство (не к другим узлам)
- **Интенсивность:** Зависит от `directions.burned.count`

**Важно:** Все три направления **независимы** и **одновременны**.

**Запрещено:**
- Показывать directions как стрелки с числами
- Делать "баланс" (given - received)
- Агрегировать в "вес" или "карму"

### Field → атмосфера пространства

**Источник:** `field`

**Визуализация:**

#### connections (sparse|woven|dense)
- **sparse:** Узлы далеко друг от друга, связи редкие
- **woven:** Узлы образуют кластеры, связи переплетаются
- **dense:** Узлы близко, связи образуют плотную ткань

**Не меняет размер узлов, только расстояние и плотность.**

#### activity (quiet|active|resonant)
- **quiet:** Пространство спокойное, движение медленное
- **active:** Пространство оживлённое, импульсы движутся
- **resonant:** Пространство вибрирует, импульсы резонируют

**Визуализация:**
- Фоновая анимация (волны, рябь, вибрация)
- Скорость движения частиц
- Частота пульсации узлов

**Запрещено:**
- Показывать field как текст или индикатор
- Делать "уровень активности" как прогресс-бар

## Двойная природа: ЛИЦО и ОБЛИК

**Дата фиксации:** 2026-05-06  
**Статус:** Canonical principle

### Философия разделения

Система обладает **двойной природой**:

- **ЛИЦО (face)** — временное движение, текущая активность, импульсы
- **ОБЛИК (shape)** — устойчивая форма, геометрия связей, структура

**Ключевое наблюдение:** После сгорания (BURN) ЛИЦО обнуляется, но ОБЛИК остаётся.

### ЛИЦО — временное движение

**Источник:** `trace.face`

**Структура:**
```json
{
  "phase": "alive|cooling|silent",
  "intensity": "present|fading|absent"
}
```

**Фазы:**
- `alive` — активное присутствие (silence < 1h)
- `cooling` — успокоение (1h < silence < 24h)
- `silent` — тишина (silence > 24h)

**Интенсивность:**
- `present` — движение видно (яркая пульсация, дрейф узлов)
- `fading` — движение затухает (слабая пульсация)
- `absent` — движение остановилось (статичные узлы)

**Визуализация ЛИЦА:**
- **Пульсация узлов** — зависит от `face.intensity`
- **Дрейф (jitter)** — микро-движение узлов при `intensity: present`
- **Импульсы по рёбрам** — волны света при активности
- **Фоновое дыхание** — вибрация пространства

**Правило:** ЛИЦО исчезает после сгорания, но не разрушает ОБЛИК.

### ОБЛИК — устойчивая форма

**Источник:** `trace.shape`

**Структура:**
```json
{
  "stability": "forming|settled",
  "persistence_hours": 168.5
}
```

**Стабильность:**
- `forming` — структура формируется (persistence < 168h / 7 дней)
- `settled` — структура устоялась (persistence >= 168h)

**Визуализация ОБЛИКА:**
- **Геометрия рёбер** — кривые Безье (статичные, без дрейфа)
- **Постоянное свечение** — базовая прозрачность узлов и рёбер
- **Расстояния между узлами** — не меняются при затухании ЛИЦА
- **Толщина связей** — зависит от количества refs, не от активности

**Правило:** ОБЛИК держится независимо от ЛИЦА. Это память системы.

### Burn Echo — эхо сгорания

**Источник:** `trace.burn_echo`

**Структура:**
```json
{
  "active": true,
  "phase": "release"
}
```

**Поведение:**
- Активируется на 1 час после работы Метронома
- Визуализируется как краткий импульс расслабления поля
- Фоновое свечение усиливается и затухает (волна)

**Визуализация:**
```javascript
if (state.trace.burn_echo?.active) {
  burnEchoEffect = Math.sin(t * 0.1) * 0.15;
}
```

**Смысл:** Система показывает момент выдоха. Это не событие, а состояние.

### Разделение слоёв в коде

**ЛИЦО (анимация):**
```javascript
// Пульсация зависит от face.intensity
let pulse = 1.0;
if (state.trace.face?.intensity === 'present') {
  pulse = 1 + Math.sin(t * 0.05) * 0.15;
}

// Дрейф узлов
let jitter = { x: 0, y: 0 };
if (state.trace.face?.intensity === 'present') {
  jitter.x = Math.sin(t * 0.03) * 2;
  jitter.y = Math.cos(t * 0.03) * 2;
}

// Импульсы по рёбрам
let faceImpulse = 0;
if (state.trace.face?.intensity === 'present') {
  faceImpulse = Math.sin(t * 0.05) * 0.2;
}
```

**ОБЛИК (геометрия):**
```javascript
// Базовая прозрачность (постоянная)
const baseOpacity = 0.4;

// Статичная геометрия рёбер (без дрейфа)
const mx = (pa.x + pb.x) / 2;
const my = (pa.y + pb.y) / 2;

// Постоянное свечение
const shapeGlow = 0.02;
```

### Граница канона

**Что НЕ является ЛИКОМ или ОБЛИКОМ:**
- ❌ Рейтинг участников — это не движение и не форма
- ❌ Метрики графа (degree, centrality) — это не геометрия
- ❌ Агрегированные показатели (karma, score) — это не присутствие

**Что ЕСТЬ ЛИЦО и ОБЛИК:**
- ✅ ЛИЦО — текущая активность, импульсы, дыхание
- ✅ ОБЛИК — устойчивая структура, память связей
- ✅ Burn Echo — момент выдоха, переход между состояниями

**Ключевой принцип:** ЛИЦО уходит, ОБЛИК остаётся. Система умеет успокаиваться без внешнего вмешательства.

## Протокол ::про.3.КОЛ:: — Окончательная фиксация двойной природы

**Дата фиксации:** 2026-05-07  
**Статус:** Canonical protocol

### Философия протокола

Протокол ::про.3.КОЛ:: (Контур Облика и Лица) устанавливает окончательные правила разделения временного и устойчивого в системе.

Это не просто визуальный эффект — это **архитектурный стандарт**, где каждое состояние имеет чёткие границы и смысл.

### Состояния ЛИЦА (face_window)

**Источник:** `trace.face_window`

**Значения:**
- `alive` — активное присутствие (silence < 4h)
- `cooling` — затухание импульса (4h < silence < 28h)
- `silent` — тишина после Метронома (silence > 28h)

**Логика определения:**
```javascript
if (silence_hours < 4) face_window = 'alive';
else if (silence_hours < 28) face_window = 'cooling';
else face_window = 'silent';
```

**Визуализация:**

**alive:**
- Высокочастотная пульсация узлов (pulse * 0.2, speed 0.1)
- Яркие импульсы по рёбрам (faceImpulse * 0.25)
- Активный дрейф узлов (jitter ±3px)
- Интенсивное фоновое дыхание (intensity 0.12)

**cooling:**
- Замедленная пульсация (pulse * 0.1, speed 0.04)
- Слабые импульсы по рёбрам (faceImpulse * 0.12)
- Минимальный дрейф (jitter ±1.5px)
- Мягкое фоновое дыхание (intensity 0.06)

**silent:**
- Статичные узлы (pulse = 1.0, jitter = 0)
- Импульсы отсутствуют (faceImpulse = 0)
- Минимальное фоновое дыхание (intensity 0.02)

### Состояния ОБЛИКА (form_hint)

**Источник:** `trace.form_hint`

**Значения:**
- `emerging` — структура формируется (< 3 подтверждённых рёбер)
- `steady` — устойчивая структура (повторяющиеся связи между О.К.)
- `settled` — кристаллизованная структура (граф не менялся > 120h)

**Логика определения:**
```javascript
// Подсчёт подтверждённых рёбер (RECOGNITION, FLOW)
const confirmed_edges = COUNT(DISTINCT edge_id WHERE edge_type IN ('RECOGNITION', 'FLOW'));

// Подсчёт повторяющихся связей
const repeating_connections = COUNT(connection WHERE COUNT(*) > 1);

// Время стабильности структуры
const structure_stable_hours = NOW() - MAX(edge.created_at);

if (confirmed_edges < 3) form_hint = 'emerging';
else if (confirmed_edges >= 3 && repeating_connections > 0) form_hint = 'steady';
else if (structure_stable_hours >= 120 && confirmed_edges >= 3) form_hint = 'settled';
```

**Визуализация:**

**emerging:**
- Свободное движение узлов (attraction = 0.01)
- Нормальное затухание (damping = 0.85)

**steady:**
- Усиленное притяжение по рёбрам (attraction = 0.015)
- Нормальное затухание (damping = 0.85)

**settled:**
- Кристаллизация — узлы стремятся к фиксированным координатам
- Усиленное притяжение (attraction = 0.03)
- Усиленное затухание (damping = 0.7) — минимизация дрейфа
- Целевые координаты (targetX, targetY) фиксируются при первом settled

### Абсолютный ноль (::0::)

**Условие:** О.К. не имеет ни одного акта в `acts_log`

**Ответ API:**
```json
{
  "state": "absolute_zero",
  "nodes": [],
  "edges": [],
  "acts": [],
  "trace": {
    "face_window": "silent",
    "form_hint": "emerging",
    "burn_echo": { "active": false }
  }
}
```

**Визуализация:**
- Пустое тёмное поле (rgba(10,10,15,1))
- Надпись `::0::` в центре (rgba(255,255,255,0.1))
- Нет узлов, рёбер, анимации

**Смысл:** Это не "отсутствие данных" — это **состояние до первого акта**. Потенциальность, ещё не проявленная.

### Этические ограничения (СТРОГО)

**Запрещено:**
- ❌ Добавлять счётчики актов (COUNT(*) на экране)
- ❌ Показывать таймлайны или хронологию
- ❌ Индикаторы "прогресса" или "уровня"
- ❌ Сравнение О.К. между собой
- ❌ Метрики производительности (acts/day, response_time)
- ❌ Рейтинги, топы, лидерборды

**Разрешено:**
- ✅ Качественные состояния (alive/cooling/silent, emerging/steady/settled)
- ✅ Геометрия связей (форма, не количество)
- ✅ Ритм дыхания (пульсация, импульсы)
- ✅ Атмосфера поля (connections, activity)

**Принцип:** Система остаётся **Зеркалом**, которое даёт ощущение "узора жизни", а не отчёт о производительности.

### Граница канона

**Что НЕ является частью протокола:**
- ❌ Аналитика поведения (кто с кем чаще взаимодействует)
- ❌ Предсказания (кто скоро станет активным)
- ❌ Рекомендации (с кем стоит связаться)
- ❌ Оценки (кто "лучше" или "хуже")

**Что ЕСТЬ часть протокола:**
- ✅ face_window — временное движение (alive/cooling/silent)
- ✅ form_hint — устойчивая форма (emerging/steady/settled)
- ✅ absolute_zero — состояние до первого акта
- ✅ Этические ограничения — защита от превращения в метрики

**Ключевая мысль:** Протокол ::про.3.КОЛ:: превращает наблюдение за системой в **архитектурный стандарт**. Это не просто визуализация — это способ различать временное и устойчивое на уровне физики данных.

---

### Что НИКОГДА не показывается

- ❌ COUNT(*) как число на экране
- ❌ SUM(intensity) как балл
- ❌ Рейтинг участников (топ-10, лидеры)
- ❌ Сравнение узлов (кто больше/меньше)
- ❌ Метрики графа (degree, centrality, betweenness)
- ❌ Агрегированные показатели (total_score, karma, reputation)

### Что показывается

- ✅ Форма присутствия (кто здесь, как давно)
- ✅ Геометрия связей (как связаны, не сколько)
- ✅ Ритм дыхания (эмиссия → передача → признание → сгорание)
- ✅ Поле смысла (где утолщается признание)
- ✅ Атмосфера пространства (connections, activity)

## Древо Инициаторов (ro.DAG визуализация)

### Философия

**Древо Инициаторов** — это геометрия доверия, а не рейтинг.

Визуализация показывает **форму связей**, а не **количество признаний**.

### Три состояния узла

#### 1. Узел вне Древа (received_um = false)

**Условие:**
```js
ro_dag_status.received_um === false
```

**Визуальные характеристики:**
- Сниженная яркость: `baseOpacity × 0.6`
- Холодный оттенок: `rgba(180,190,210,${opacity})`
- Тонкий контур: `strokeStyle rgba(180,190,210,${opacity * 0.5})`, `lineWidth 1`
- Усиленный drift: `driftMultiplier = 1.5`

**Смысл:** Состояние потенциала. Узел ещё не получил признания, но присутствует в поле.

#### 2. Узел внутри Древа (received_um = true, sent_ue = false)

**Условие:**
```js
ro_dag_status.received_um === true && ro_dag_status.sent_ue === false
```

**Визуальные характеристики:**
- Устойчивое свечение: `baseOpacity × 1.1`
- Нейтральный оттенок: `rgba(200,200,220,${opacity})`
- Стабильная орбита: `driftMultiplier = 1.0`
- Связи отображаются чётче

**Смысл:** О.К. получил признание, но ещё не передал импульс. Редкое состояние.

#### 3. Узел-инициатор (in_tree = true)

**Условие:**
```js
ro_dag_status.received_um === true && ro_dag_status.sent_ue === true
```

**Визуальные характеристики:**
- Выраженное присутствие: `baseOpacity × 1.25`
- Увеличенный размер: `radius × 1.15`
- Тёплый оттенок: `rgba(220,210,200,${opacity})`
- Мягкий ореол участия: `glowRadius × 2.5`, `glowIntensity 0.22`
- Связи слегка утолщаются

**Смысл:** Полноценное участие в Древе. Это не статус превосходства, а фиксация участия в распространении признания.

### Органическая геометрия

**Используется:**
- Кривые Безье для связей
- Мягкое притяжение между узлами
- Постепенное уплотнение ядра
- Drift по периферии для узлов вне Древа

**Запрещено:**
- ❌ Древовидные UI-диаграммы
- ❌ Иерархические схемы
- ❌ Графики "уровней"
- ❌ Центральные "хабы"
- ❌ Визуальное доминирование

### Метафора визуализации

Древо должно ощущаться как:
- **Живая ткань** — связи утолщаются и истончаются
- **Крона** — ветви расходятся от ядра
- **Распределённая память признания** — каждая связь хранит след
